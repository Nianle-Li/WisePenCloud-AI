# 前沿探索功能设计方案

## 一、设计目标与整体思路

本功能在现有 WisePen Chat Service 的基础上，扩展出**子 Agent 系统**。核心目标是让主 Agent 在一轮对话内可以按需创建并调用独立的子 Agent，子 Agent 拥有独立的 Action Loop、隔离的消息历史，且支持多个子 Agent 并行执行。

整体设计遵循以下原则：

- **最大化复用现有基础设施**：`QueryLoopRuntime`（Action Loop）、`ToolRegistry.derive()`（权限模型）、Redis/MongoDB（存储）、Ephemeral 机制（历史隔离）均直接复用，不重复造轮子
- **统一配置类 `AgentConfig`**：主 Agent 会话配置与子 Agent 配置使用同一类型，三种使用场景（主 Agent 会话 / 临时子 Agent / 预制模板）共用同一套字段语义；子 Agent 创建时若未提供 config，`create_sub_agent` 工具默认将 `disable_skill_matching` 设为 `True`
- **参考 Claude Code 的设计模式**：`SubAgentRuntime` 对应 CC 的 `runAgent()` 独立生成器实例；子 Agent 消息历史采用 fresh 模式（对应 CC 的 `forkContextMessages=[]`），每次调用只携带 prior_turns 摘要 + 当前 query，不传入父对话历史

### 2.1 整体分层与模块关系

### 2.2 子 Agent 完整生命周期

一次典型的子 Agent 调用流程如下：

```
主 Agent LLM 决定使用子 Agent
        │
        ▼
[1] call create_sub_agent(description?, display_name?, template_id?, config?: AgentConfig)
        │  解析参数（template 为基础，config 字段补丁覆盖；详见 §5.1）
        │  生成 agent_id，将 {display_name（未提供时取 description）, AgentConfig} 写入 Redis（TTL 24h）
        │  返回 { agent_id: "sa_a1b2c3d4" }
        │
        │  ※ agent_id 仅在当前对话轮次内有效，工具描述中约定 LLM 不应跨轮次复用；
        │    TTL 24h 是服务端兜底，保证同一轮内多次 call 的可靠性
        │
        ▼
[2] call call_sub_agent(agent_id, query)
        │
        ├─ 从 Redis 读取 AgentConfig
        ├─ 从 Redis 读取多轮调用历史（若有）
        │
        ├─ 创建 SubAgentRuntime 实例，执行 run(query)
        │       ├─ 推送 SubAgentStartEvent → SSE
        │       ├─ 内部新建独立的 QueryLoopRuntime 实例
        │       │       │ 执行完整 ReAct Action Loop
        │       │       │ 每个 TextDeltaEvent 推送 SubAgentStepEvent → SSE
        │       │       └─ 返回最终文本 result
        │       ├─ 推送 SubAgentCompleteEvent → SSE
        │       └─ 返回 result 给 CallSubAgentTool
        │
        ├─ 将 (query, result) 摘要对写入 Redis 历史（RPUSH + LTRIM 20）
        │
        └─ 返回 result 给主 Agent（tool result，is_ephemeral=False，完整持久化）

【持久化】
    ChatTurnFinalizer 将 call_sub_agent tool result 原文写入 Redis/MongoDB（不做 redact）
    SubAgentRuntime 内部消息历史随 run() 返回后自动 GC（不写入任何持久化存储）
```

### 2.3 多子 Agent 并行执行

当主 Agent LLM 在同一步骤中发出多个 `call_sub_agent` 工具调用时，`QueryLoopRuntime` 的现有 `asyncio.gather` 机制天然支持并行：

```
主 Agent 一步内调用 3 个工具：

asyncio.gather(
    call_sub_agent(agent_a, "分析安全漏洞"),   ─── SubAgentRuntime A 独立运行 ───┐
    call_sub_agent(agent_b, "生成测试用例"),   ─── SubAgentRuntime B 独立运行 ───┤─→ 全部完成后汇总返回给主 Agent
    call_sub_agent(agent_c, "搜索文档"),       ─── SubAgentRuntime C 独立运行 ───┘
)

SSE 流（同一连接，3个子 Agent 的进度交织输出）：
  {"type":"sub_agent_start",  "agent_id":"agent_a", ...}
  {"type":"sub_agent_start",  "agent_id":"agent_b", ...}
  {"type":"sub_agent_step",   "agent_id":"agent_a", "delta":"正在扫描..."}
  {"type":"sub_agent_step",   "agent_id":"agent_b", "delta":"正在生成..."}
  {"type":"sub_agent_complete","agent_id":"agent_c", ...}
  ...
```

SSE 多路复用通过 `tool_context["sse_sender"]` 实现：`ChatTurnCoordinator` 在构造 `tool_context` 时注入当前请求的 SSE 写入器，所有并行的 `CallSubAgentTool` 实例共享同一写入器，用 `agent_id` 区分来源。

## 三、配置模型设计：AgentConfig

### 3.1 AgentConfig 字段结构

`AgentConfig` 是主 Agent 会话配置与子 Agent 配置的**统一类型**，三种使用场景共用同一套字段：

| 使用场景                                  | 典型配置                                                                                |
| ----------------------------------------- | --------------------------------------------------------------------------------------- |
| 主 Agent 会话配置（存于 `ChatSession`）   | `enable_long_term_memory=True`，`max_iterations=20`                                     |
| 临时子 Agent（LLM 在对话中动态创建）      | 通常使用全默认值；`create_sub_agent` 未收到 config 时默认 `disable_skill_matching=True` |
| 预制子 Agent 模板（存于 `AgentTemplate`） | 固定 `allow_tool_names`，专用 `system_prompt_override`                                  |

`AgentConfig` 字段按功能分为四组：

```
AgentConfig
├── 模型控制
│     model_id: Optional[str]              # None = 子 Agent 时继承主 Agent 模型，主 Agent 时使用系统默认
│
├── 提示词控制（两种模式）
│     system_prompt_override: Optional[str] # 完全替换，适合专用子 Agent
│     system_prompt_append:   Optional[str] # 在默认后追加，适合轻度扩展
│
├── 工具权限控制（映射到 ToolRegistry.derive() 的同名参数）
│     allow_tool_names:            白名单，None = 继承全部
│     deny_tool_names:             黑名单，None = 不屏蔽
│     expose_reserved_tool_names:  解禁 reserved 工具
│
├── Skill 控制
│     allowed_skill_ids:    指定可匹配的 Skill 范围
│     disable_skill_matching: bool = False  # create_sub_agent 未提供 config 时在工具内覆盖为 True
│
└── 运行时策略
      max_iterations: int = 10              # 防止无限循环
      enable_long_term_memory: bool = False # 是否启用 Mem0 长期记忆检索

# display_name 由 create_sub_agent(display_name?) 参数传入，随 AgentConfig 一并存入 Redis
```

### 3.2 子 Agent 消息列表构建（fresh 模式）

`SubAgentRuntime` 为每次 `call_sub_agent` 调用构造 `initial_messages`，对应 Claude Code 的 `forkContextMessages=[]`（fresh 模式）：

```
initial_messages = [
    # 若本次调用前存在历史轮次，展开 prior_turns（子 Agent 视角的记忆前缀）
    {"role": "user",      "content": "<第1次 query>"},
    {"role": "assistant", "content": "<第1次 result>"},
    ...                                                    # 更多历史轮次（按时序）
    # 本次调用的 query
    {"role": "user",      "content": query},
]
```

### 3.3 主 Agent 默认配置

`AgentConfig` 的 Pydantic 字段默认值倾向于保守策略（`enable_long_term_memory=False`，`max_iterations=10`），主 Agent 使用单独的常量明确设置所需值（见 §7.1 代码）：

```python
DEFAULT_MAIN_AGENT_CONFIG = AgentConfig(
    enable_long_term_memory=True,   # 主 Agent 需要 Mem0 长期记忆
    max_iterations=20,              # 主 Agent 允许更多推理步骤
)
```

`ChatSession.agent_config` 为 `None` 时使用此常量。

## 四、子 Agent 历史隔离机制

### 4.1 方案：两层隔离

WisePen 是有状态服务，`ChatTurnFinalizer` 会将所有消息持久化到 Redis 和 MongoDB，需要**主动设计隔离**（CC 单进程工具的局部变量 GC 在这里不适用）。

**第一层：SubAgentRuntime 内部历史（运行时隔离）**

`SubAgentRuntime.run()` 内维护的消息列表是函数局部变量，子 Agent 完整推理过程（所有 LLM 消息、工具调用记录）不流出此作用域，函数返回后自动 GC。

**第二层：直接持久化（完整保留 tool result）**

`CallSubAgentTool` 的工具执行结果以 `is_ephemeral=False` 标记返回（即 `ToolResult(content=result, is_ephemeral=False)`）。`ChatTurnFinalizer` 在调用 `_redact_ephemeral()` 清理临时内容时，不会处理该条 tool result——其完整内容原文写入 Redis/MongoDB，不做任何截断或置空：

```python
# CallSubAgentTool.execute() 返回示意
return ToolResult(content=result, is_ephemeral=False)  # 完整内容，不被 _redact_ephemeral() 处理
```

```

主 Agent 视角的消息历史（运行时）：
assistant: [tool_use: call_sub_agent(agent_a, "分析安全漏洞")]
tool: [tool_result(is_ephemeral=False): "发现3个XSS漏洞：...(完整2000字报告)"] ← 完整可见

                    ↓ ChatTurnFinalizer 持久化（_redact_ephemeral() 跳过此条）

主 Agent 视角的消息历史（持久化后，Redis + MongoDB）：
assistant: [tool_use: call_sub_agent(agent_a, "分析安全漏洞")]
tool: [tool_result: "发现3个XSS漏洞：...(完整2000字报告)"] ← 原文保留
assistant: "根据安全分析结果，发现以下问题..." ← 主 Agent 综合文本保留

```

> 注：上下文 token 膨胀问题由对话历史统一截断/压缩策略管理，不在工具层处理。

### 4.2 跨调用历史（同一子 Agent 被多次 call）

同一子 Agent 在一轮对话中可被多次调用，需区分两种层次的历史：

| 历史类型                  | 内容                                                                     | 处理方式                                                                  |
| ------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------- |
| **子 Agent 内部执行历史** | 子 Agent 在一次 `call_sub_agent` 调用内部产生的所有 LLM 消息、工具调用链 | 不保留——运行时 GC（`SubAgentRuntime.run()` 局部变量，函数返回后自动释放） |
| **调用交流摘要对**        | 每次 `call_sub_agent` 的 query 和 result 摘要对                          | 保留——写入 Redis List，用于下次调用提供连续性上下文                       |

跨调用历史存于 Redis List，TTL 24h，最多 20 轮：

```python
# 写入：每次 call_sub_agent 完成后（子 Agent 视角的摘要对）
entry = json.dumps({"query": query, "result": result})
await redis.rpush(f"wisepen:sub_agent_ctx:{session_id}:{agent_id}", entry)
await redis.ltrim(f"wisepen:sub_agent_ctx:{session_id}:{agent_id}", -20, -1)  # 保留最近20次调用
await redis.expire(f"wisepen:sub_agent_ctx:{session_id}:{agent_id}", 86400)

# 读取：下次调用开始时展开为 LLM 消息对（子 Agent 视角）
prior_turns = []
for entry_json in await redis.lrange(key, 0, -1):
    e = json.loads(entry_json)
    prior_turns += [
        {"role": "user",      "content": e["query"]},   # 上次被问了什么
        {"role": "assistant", "content": e["result"]},  # 上次回答了什么
    ]
```

读取后按时序展开为 user/assistant 消息对，拼接到 `initial_messages` 前缀（见 §3.2）。

### 4.3 SubAgentRuntime 构建细节

**工具集派生**

`SubAgentRuntime` 从**工具注册表（`tool_registry`）** 而非父 ToolScope 快照做 derive，保证可预测的权限基线：

```python
tool_scope = tool_registry.derive(
    allow_tool_name_set=set(agent_config.allow_tool_names) if agent_config.allow_tool_names else None,
    deny_tool_name_set=(set(agent_config.deny_tool_names or []))
                       | {"create_sub_agent", "call_sub_agent"},  # 强制屏蔽，防止递归
    expose_tool_name_set=set(agent_config.expose_reserved_tool_names or []),
)
```

**SSE 事件转发（内聚在 SubAgentRuntime）**

`SubAgentRuntime` 持有 `sse_sender` 和 `agent_id`，内部监听 `QueryLoopRuntime` 输出并转发；`CallSubAgentTool` 只负责参数准备和最终结果收集：

```
SubAgentRuntime(config, display_name, sse_sender, agent_id).run(query)
    │
    ├─ 推送 SubAgentStartEvent(agent_id, display_name)
    │
    ├─ QueryLoopRuntime 内部循环（流式生成）
    │       │ 每产出一个 TextDeltaEvent(delta)
    │       ▼
    │   SubAgentRuntime 包装 → 推送 SubAgentStepEvent(agent_id, delta)
    │
    └─ 循环结束，累积最终 result 文本
        ├─ 推送 SubAgentCompleteEvent(agent_id, result_summary≤500字, success=True)
        └─ 返回完整 result 给 CallSubAgentTool（不截断，主 Agent 使用）
```

## 五、工具权限与激活设计

### 5.1 create/call_sub_agent 工具的可见性控制

这两个工具标记为 `reserved=True`，默认对 LLM 不可见。系统支持两种模式解禁这两个工具：

**模式 A：预制模式（AgentTemplate 全量加载）**

`SkillAndAgentMatcher` 每轮固定返回全部已启用的 `AgentTemplate`（不做关键词匹配，永真），`ChatTurnCoordinator` 自动在 `ToolRegistry.derive()` 的 `expose_tool_name_set` 中解禁：

```

用户发起任意对话
│
▼
SkillAndAgentMatcher.match()
├─ Skill 匹配: 正常关键词匹配
└─ AgentTemplate: 直接返回全部已启用模板（无关键词过滤）
│
▼
expose_set |= {"create_sub_agent", "call_sub_agent"}
MatchResult.agent_templates 传入 ChatContextAssembler
ChatContextAssembler 在 System Prompt 追加可用子 Agent 清单
│
▼
LLM 感知到可用工具及模板清单，自主决定是否调用 create_sub_agent(template_id=...)

```

**模式 B：自由模式（用户手动解禁）**

用户通过 API 将 `create_sub_agent` 和 `call_sub_agent` 加入会话的 `expose_reserved_tool_names`，LLM 可在任意对话轮次中自主决定是否创建子 Agent，无需匹配 AgentTemplate：

```

PATCH /sessions/{id}/agent-config
body: { "expose_reserved_tool_names": ["create_sub_agent", "call_sub_agent"] }

→ ChatTurnCoordinator 的 expose_set 每轮均包含这两个工具
→ LLM 可自主调用 create_sub_agent(config={...})，config 中直接内联 AgentConfig
→ template_id 与 config 均为可选参数，同时提供时以 template 为基础、config 字段逐项覆盖（补丁语义）

```

两种模式共用同一套工具实现；`template_id` 和 `config` 均不提供时使用 `AgentConfig` 全默认值。

### 5.2 子 Agent 的工具权限继承与限制

`SubAgentRuntime` 构造 `tool_scope` 时的权限逻辑：

```
子 Agent 工具集 = ToolRegistry 全部非 reserved 工具
    - deny_tool_names（黑名单，来自 AgentConfig）
    - {create_sub_agent, call_sub_agent}（强制屏蔽，防止嵌套递归）
    ↓ 若配置了 allow_tool_names：
    只保留白名单内的工具（否则继承注册表全部非 reserved 工具）
```

未配置 `allow_tool_names` 时子 Agent 从**注册表**派生全部非 reserved 工具，从零开始，不受父 Agent 的 deny/allow 配置约束。这是需求⑨"剩下未涉及的部分 model 自主决策"的实现：默认最大工具集，用 deny 显式关闭，而非用 allow 逐一开放。

## 六、预制子 Agent（AgentTemplate）

### 6.1 设计：复用 Skill 数据模式

`AgentTemplate` 与现有 `Skill` 文档的结构相似，区别在于 Skill 携带的是指令文档（由 LLM 加载阅读），而 AgentTemplate 携带的是 `AgentConfig`（直接驱动独立的 Action Loop）：

```

Skill 文档：
skill_id, title, keywords[], content_markdown, asset_files[]
→ LLM 通过 load_skill 工具读取文本，在自身上下文中参考执行

AgentTemplate：
template_id, display_name, description
agent_config: AgentConfig { system_prompt_override, allow_tool_names, model_id, ... }
→ LLM 通过 create_sub_agent(template_id) + call_sub_agent 委托给独立运行时执行
```

Skill 通过关键词触发激活；AgentTemplate 则全量常驻于 System Prompt 的可用清单中，由 LLM 自主决定是否使用。二者执行方式不同：Skill 是主 Agent 自身的能力增强，AgentTemplate 是委托给独立运行时实例执行。

> **为什么 create 和 call 分成两步？** 拆分的核心原因是支持**同一 `agent_id` 在一轮内多次 `call`**：create 只做一次，agent_id 绑定到 AgentConfig（含 system_prompt、工具集等），后续每次 call 都携带递增的 prior_turns，形成连续性上下文。若合并为一步，每次调用都需要重新传入完整配置，既冗余又无法复用跨调用历史。这与 CC 合并为单步 `AgentTool` 的做法不同，是 WisePen 有状态服务场景下的有意差异。

### 6.2 全量加载流程

扩展现有 `KeywordSkillMatcher` 为 `SkillAndAgentMatcher`，在同一次匹配中同时处理 Skill 和 AgentTemplate，结果合并到统一的 `MatchResult`：

```python
@dataclass
class MatchResult:
    skills: List[SkillMeta]
    agent_templates: List[AgentTemplateMeta]   # 新增字段
```

`AgentTemplateMeta` 是 `AgentTemplate` 的轻量投影（仅含 `template_id`、`display_name`、`description`，不含完整 `agent_config`），用于向 LLM 呈现可用模板清单。AgentTemplate **不做关键词过滤**，`match()` 方法每次直接返回全部已启用模板（`enabled=True`）。`ChatTurnCoordinator` 根据 `template_id` 另行从 MongoDB 读取完整 `AgentTemplate`。

AgentTemplate 的缓存刷新机制与 Skill 相同：启动时 eager warmup，后台定时刷新（复用 `SkillCacheRefresher` 模式）。

| `wisepen:sub_agent_config:{session_id}:{agent_id}` | String | 86400s | —          | `CreateSubAgentTool.execute()`，存 `{display_name, ...AgentConfig}` | `CallSubAgentTool.execute()` |
| `wisepen:sub_agent_ctx:{session_id}:{agent_id}`    | List   | 86400s | 20轮 LTRIM | `call_sub_agent` 执行完毕                                           | `call_sub_agent` 开始时      |

## 八、新增 SSE 事件类型

在现有 `StreamEvent` 体系中追加三个子 Agent 专用事件，`vercel_sse_mapper.py` 同步增加序列化映射：

| 事件类                  | 触发时机                       | 关键字段                                                               |
| ----------------------- | ------------------------------ | ---------------------------------------------------------------------- |
| `SubAgentStartEvent`    | `call_sub_agent` 开始执行      | `agent_id`, `display_name`                                             |
| `SubAgentStepEvent`     | 子 Agent 产出 `TextDeltaEvent` | `agent_id`, `delta`                                                    |
| `SubAgentCompleteEvent` | 子 Agent 完成或失败            | `agent_id`, `result_summary`（截断至 500 字的进度展示摘要）, `success` |
